# Calculation Fixes

## 2026-06-07 — первая миля Москва-Москва считается по маршрутной таблице

Ошибка:
Маршрут `Москва → Москва` сначала был ошибочно истолкован как отсутствие первой мили. Это давало `0 руб./шт`, хотя в Google Sheets для первой мили есть отдельная стоимость маршрута `Москва → Москва = 1675 руб./паллет`.

Как должно быть:
Первая миля всегда берётся из маршрутной таблицы `откуда → куда`, включая маршруты с одинаковым городом отправки и доставки. Итоговая матрица показывает стоимость в `руб./шт`, поэтому ставка за паллету делится на `itemsPerPallet`.

Что изменено:
- `src/lib/calculator.ts`: сохранён расчёт `routeRubPerPallet / itemsPerPallet`; в детализации добавлена формула `руб./паллет / шт.`.
- `preview/app.js`: та же подпись добавлена в статический прототип.
- `tests/calculator.test.ts`: закреплён контрольный пример `Москва → Москва`.
- `docs/tariff-validation.md`: блок первой мили помечен как подтверждённый бизнесом.

Проверка:
Для SKU `Вешалки`, `itemsPerPallet = 30`: `1675 / 30 = 55,83 руб./шт` без НДС. В режиме `с НДС`: `55,83 * 1,22 = 68,12 руб./шт`.

## 2026-06-03 — WB FBO delivery tariff source

Ошибка:
WB FBO использовал общий тариф коробов `/api/v1/tariffs/box` для логистики до покупателя. Для склада `Коледино` на `2026-06-03` это давало `89,70` за первый литр и `27,30` за следующий литр, коэффициент `195%`.

Как должно быть:
WB FBO должен использовать тариф доставки из `/api/tariffs/v1/acceptance/coefficients` на дату поставки/расчета. Для склада `Коледино`, тип поставки `Короб`, дата `2026-06-03`: `94,30` за первый литр и `28,70` за следующий литр, коэффициент `205%`.

Что изменено:
- `src/lib/calculator.ts`: FBO-доставка WB теперь сначала берет `acceptance[тип].deliveryBaseLiterRub` и `acceptance[тип].deliveryAdditionalLiterRub`.
- `preview/app.js`: та же логика применена в прототипе на `http://127.0.0.1:4173`.
- `tests/calculator.test.ts`: ожидаемая сумма FBO-доставки обновлена.

Проверка:
Для SKU `Вешалки`, объем `41,944 л`, склад `Коледино`, тип `Короб`: `94,30 + (41,944 - 1) * 28,70 = 1 269,39 руб.`. FBS остается на marketplace-тарифе `89,70 + (41,944 - 1) * 27,30 = 1 207,47 руб.`.

## 2026-06-03 — WB FBO unavailable acceptance should not hide total

Ошибка:
Если `/api/tariffs/v1/acceptance/coefficients` возвращал недоступную приемку на дату расчета, калькулятор оставлял статью `Приёмка WB = 0`, но вся схема становилась неполной и в карточке результата показывалась надпись `Тариф не найден`.

Как должно быть:
Недоступная приемка WB является неблокирующим предупреждением. В статье расхода остается `0`, предупреждение показывается в карточке, но итоговая сумма расходов по схеме все равно отображается.

Что изменено:
- `src/lib/calculator.ts`: предупреждение `Поставка WB на ... недоступна` больше не делает схему неполной.
- `preview/app.js`: то же поведение добавлено в прототип.
- `tests/calculator.test.ts`: зафиксировано, что WB FBO для примера `Коледино / Короб` показывает итог `2 512,72 руб.` при предупреждении о недоступной приемке.

## 2026-06-03 — WB FBO storage tariff source

Ошибка:
WB FBO использовал общий тариф хранения коробов `/api/v1/tariffs/box`. Для склада `Коледино` на `2026-06-03` это давало `0,10 / 0,10` в день.

Как должно быть:
WB FBO должен использовать хранение из `/api/tariffs/v1/acceptance/coefficients` на дату поставки/расчета. Для склада `Коледино`, тип поставки `Короб`, дата `2026-06-03`: `0,16` за первый литр в день и `0,16` за следующий литр в день, коэффициент `200%`.

Что изменено:
- `src/lib/calculator.ts`: FBO-хранение WB теперь сначала берет `acceptance[тип].storageBaseLiterRub` и `acceptance[тип].storageAdditionalLiterRub`.
- `preview/app.js`: та же логика применена в прототипе.
- `tests/calculator.test.ts`: ожидаемая сумма хранения WB обновлена.

Проверка:
Для SKU `Вешалки`, объем `41,944 л`, склад `Коледино`, тип `Короб`, `30` дней: `(0,16 + (41,944 - 1) * 0,16) * 30 = 201,33 руб.`.

## 2026-06-11 — Ozon FBO storage free placement days

Ошибка:
Ozon FBO раньше не учитывал хранение маркетплейса. Для КГТ-товаров это особенно заметно: у SKU `Столик`, тип Ozon `Стол обеденный`, бесплатный срок по справочнику составляет `30` дней для КГТ.

Как должно быть:
Хранение Ozon применяется только в FBO. Бесплатные дни берутся из `Озон_Сроки_бесплатного_размещения_010626_1778767885.xlsx`, лист `Прайс размещение (БЗ)`, по категории и типу товара. После бесплатного срока считаем `max(0, storageDays - freeDays) * литры * ставка`, где стандартный товар = `2,50 руб./л/день`, КГТ = `0,10 руб./л/день`.

Что изменено:
- `scripts/normalize_tariffs.py`: добавлен импорт справочника бесплатных дней хранения Ozon.
- `src/lib/calculator.ts`: Ozon FBO добавляет статью `Хранение Ozon FBO`; FBS и DBS эту статью не получают.
- `preview/app.js`: та же логика применена в прототипе.
- `tests/calculator.test.ts`: зафиксирован расчет для `Столик` как Ozon КГТ при `60` днях хранения.

Проверка:
Для SKU `Столик`, объем `121,275 л`, Ozon КГТ, `60` дней хранения: `(60 - 30) * 121,275 * 0,10 = 363,83 руб. с НДС`.

## 2026-06-11 — Ozon KGT FBO/FBS delivery uses the common logistics matrix

Ошибка:
Ozon КГТ для FBO/FBS показывал предупреждение, что доставка считается по стандартной матрице и нужен отдельный источник КГТ-тарифов.

Как должно быть:
По бизнес-уточнению от 2026-06-11 доставка КГТ считается по тому же принципу, что и доставка обычных товаров: по направлению и ВГХ из `logistika-fbo-fbs-01052026_1777018200.xlsx`.

Что изменено:
- `src/lib/calculator.ts`: предупреждение для Ozon КГТ в FBO/FBS убрано; расчет продолжает использовать общую тарифную матрицу.
- `preview/app.js`: то же поведение добавлено в прототип.
- `tests/calculator.test.ts`: зафиксировано, что Ozon КГТ FBO/FBS считается по общей матрице без предупреждения.

Проверка:
Для Ozon КГТ с объемом в диапазоне тарифа FBO/FBS логистика берется из той же строки матрицы, что и для обычного товара с таким же направлением и ВГХ.
