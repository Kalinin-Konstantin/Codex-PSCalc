# План админки калькулятора

Документ фиксирует поэтапный план перехода от локального прототипа калькулятора к версии с доступами, сохранением расчетов и администрированием.

## Цели

1. Ограничить доступ к калькулятору после деплоя.
2. Дать зарегистрированным пользователям возможность сохранять расчеты по селлерам.
3. Дать администратору доступ к пользователям и их расчетам.
4. Дать администратору возможность менять коммерческие настройки по умолчанию для новых расчетов.

## Этапы реализации

| Этап | Название | Что делаем | Результат этапа |
|---|---|---|---|
| 1 | Архитектура и модель данных | Выбираем способ деплоя, базу данных, авторизацию, роли и структуру сущностей `User`, `Seller`, `Calculation`, `CommercialSettingsProfile`. | Понятна техническая схема, список таблиц/API и границы MVP. |
| 2 | Авторизация и роли | Добавляем вход в систему, роли `admin` и `user`, закрываем калькулятор от неавторизованных пользователей. | Калькулятор доступен только после входа; админ и пользователь разделены по правам. |
| 3 | Сохранение расчетов | Добавляем селлеров и сохранение расчетов с текущими SKU, настройками, тарифами и результатами. | Пользователь может создать селлера, сохранить расчет и открыть его позже. |
| 4 | Админский просмотр | Добавляем админский раздел со списком пользователей, селлеров и сохраненных расчетов. | Администратор видит общую картину и может открыть расчет любого пользователя. |
| 5 | Коммерческие настройки по умолчанию | Переносим дефолтные наценки в управляемый профиль, который редактирует админ. | Новые расчеты стартуют с актуальными коммерческими настройками, заданными администратором. |
| 6 | Полировка и безопасность | Добавляем проверки доступа, базовые аудиты, понятные пустые состояния, миграции и регрессионные тесты. | Версия готова к демонстрационному деплою. |

## Бизнес-правила MVP

- Пользователь видит только своих селлеров и свои расчеты.
- Администратор видит всех пользователей, всех селлеров и все расчеты.
- Пользователь может зарегистрироваться самостоятельно, но до подтверждения администратором не получает доступ к калькулятору.
- Вход в систему делаем по email и паролю.
- Сохраненный расчет должен хранить снимок настроек на момент сохранения, чтобы старые расчеты не менялись неожиданно после обновления дефолтов.
- Изменение дефолтных наценок администратором влияет на новые расчеты, а не пересчитывает старые автоматически.
- В MVP роль `user` означает менеджера PIM.Seller, который проводит расчет и демонстрирует интерфейс селлеру. Менеджер может открыть внутреннюю коммерческую панель на главной странице и точечно менять коммерческие условия расчета.
- Селлер в MVP не входит в калькулятор самостоятельно. Для селлера готовится клиентская Excel-выгрузка.
- Клиентская Excel-выгрузка не показывает маржу, себестоимость и проценты наценки PIM.Seller.
- Селлер в MVP хранит только название.
- Один пользователь может иметь несколько селлеров; у одного селлера может быть несколько расчетов.

## Решения этапа 1

| Вопрос | Решение |
|---|---|
| Регистрация | Пользователь регистрируется самостоятельно, затем ждёт подтверждения администратора. |
| Авторизация | Email + пароль. |
| Деплой и инфраструктура | Vercel + Supabase. |
| Данные селлера | Только `name`, то есть наименование селлера. |
| Связи данных | Один пользователь может иметь несколько селлеров; один селлер может иметь несколько расчетов. |

## Предварительная модель данных

| Сущность | Назначение | Ключевые поля MVP |
|---|---|---|
| `profiles` | Профиль пользователя поверх Supabase Auth. | `id`, `email`, `role`, `status`, `created_at`, `approved_at`, `approved_by` |
| `sellers` | Селлеры пользователя. | `id`, `owner_id`, `name`, `created_at`, `updated_at` |
| `calculations` | Сохраненные расчеты по селлерам. | `id`, `seller_id`, `owner_id`, `name`, `snapshot`, `created_at`, `updated_at` |
| `commercial_settings_profiles` | Дефолтные коммерческие настройки для новых расчетов. | `id`, `name`, `settings`, `is_default`, `created_at`, `updated_at`, `updated_by` |

## Предварительные статусы пользователя

| Статус | Что означает |
|---|---|
| `pending` | Пользователь зарегистрировался, но администратор ещё не подтвердил доступ. |
| `approved` | Пользователь подтверждён и может пользоваться калькулятором. |
| `blocked` | Доступ пользователя вручную заблокирован администратором. |

## Предварительные роли

| Роль | Права MVP |
|---|---|
| `user` | Менеджер PIM.Seller: может пользоваться калькулятором, создавать селлеров, сохранять свои расчеты и открывать внутреннюю коммерческую панель на главной странице. |
| `admin` | Видит пользователей, подтверждает регистрации, видит всех селлеров и расчеты, управляет дефолтными коммерческими настройками. |

## Предварительные API/экраны

| Блок | Что нужно |
|---|---|
| Регистрация | Экран регистрации, создание Supabase Auth пользователя, профиль со статусом `pending`. |
| Ожидание подтверждения | Экран “заявка отправлена, ожидайте подтверждения”. |
| Админ: пользователи | Список пользователей со статусами, кнопки подтвердить/заблокировать. |
| Селлеры | Список селлеров текущего пользователя, создание/переименование/удаление. |
| Расчеты | Сохранить текущий расчет, открыть сохраненный расчет, список расчетов по селлеру. |
| Админ: расчеты | Просмотр расчетов всех пользователей и селлеров. |
| Админ: настройки | Редактирование дефолтного профиля коммерческих настроек. |

## Этап 2: авторизация и роли

Статус: реализована базовая заготовка.

Что добавлено:

- Supabase SSR-клиенты для Next.js.
- Proxy для обновления auth cookies.
- `.env.example` с нужными переменными окружения.
- SQL-миграция `supabase/migrations/202606220001_auth_profiles.sql`.
- Главная страница теперь закрывает калькулятор до авторизации и статуса `approved`.
- Самостоятельная регистрация создает пользователя Supabase Auth, а профиль получает статус `pending`.
- Админская страница `/admin` показывает пользователей и позволяет подтвердить или заблокировать доступ.

Как поднять:

1. Создать проект Supabase.
2. Скопировать `.env.example` в `.env.local`.
3. Заполнить `NEXT_PUBLIC_SUPABASE_URL` и `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`.
4. Выполнить SQL из `supabase/migrations/202606220001_auth_profiles.sql` в Supabase SQL editor или через Supabase CLI.
5. Зарегистрировать первого пользователя через интерфейс калькулятора.
6. Сделать его администратором вручную в Supabase SQL editor:

```sql
update public.profiles
set role = 'admin',
    status = 'approved',
    approved_at = now()
where email = 'admin@example.com';
```

После этого первый администратор сможет подтверждать остальных пользователей через `/admin`.

## Этап 3: сохранение расчетов

Статус: реализована базовая заготовка.

Что добавлено:

- SQL-миграция `supabase/migrations/202606220002_sellers_calculations.sql`.
- Таблица `sellers`: селлеры пользователя, в MVP хранится только название.
- Таблица `calculations`: сохраненные расчеты по селлеру.
- RLS-политики: пользователь видит и меняет только своих селлеров/расчеты; администратор может видеть все.
- Server actions для создания селлера и сохранения расчета.
- На главной странице добавлен блок `Селлер и сохранённые расчёты`.
- Сохраненный расчет хранит снимок `version`, `skus`, `settings`.
- Открытие сохраненного расчета идет через параметры URL `?seller=...&calculation=...`.

Текущее бизнес-решение:

- Изменение тарифных JSON-справочников в проекте влияет на новый пересчет открытого снимка.
- Снимок фиксирует введенные SKU и настройки пользователя, но не хранит полный рассчитанный результат по всем схемам.
- Если позже понадобится юридически фиксировать коммерческое предложение “как было выдано клиенту”, нужно добавить отдельный immutable-export snapshot с рассчитанными итогами и версией тарифов.

## Этап 4: админский просмотр

Статус: реализована базовая заготовка.

Что добавлено:

- Админская страница `/admin` теперь показывает не только пользователей, но и общий список сохраненных расчетов и селлеров.
- В списках расчетов и селлеров администратор видит владельца записи.
- Администратор может открыть расчет или селлера любого пользователя из `/admin`.
- Открытие чужого расчета идет через параметры URL `?owner=...&seller=...&calculation=...`.
- Главная страница умеет загружать рабочее пространство другого пользователя только для администратора.
- Если администратор открыл чужой расчет, калькулятор работает в режиме просмотра: создание селлера и сохранение расчета отключены.

Текущее бизнес-решение:

- Администратор может смотреть расчеты пользователей, но не редактирует их через основной экран калькулятора.
- Это снижает риск случайно переписать клиентский расчет во время проверки или демонстрации.
- Если позже понадобится полноценное редактирование чужих расчетов администратором, это нужно добавить отдельным осознанным действием с явной пометкой “редактирование от имени пользователя”.

## Этап 5: коммерческие настройки по умолчанию

Статус: реализована базовая заготовка.

Что добавлено:

- SQL-миграция `supabase/migrations/202606220003_commercial_settings_profiles.sql`.
- Таблица `commercial_settings_profiles` хранит управляемый администратором дефолтный профиль коммерческих настроек.
- RLS-политики: подтвержденные пользователи могут читать дефолтный профиль, изменять его может только администратор.
- В `/admin` добавлен блок `Дефолты для новых расчётов`.
- Администратор может менять:
  - наценку первой мили;
  - тип складской поставки PIM.Seller;
  - включение складских групп и наценки по ним;
  - наценки средней мили по диапазонам;
  - наценки последней мили.
- Новые расчеты стартуют с коммерческими дефолтами из Supabase.
- Сохраненные расчеты продолжают открываться из собственного snapshot и не меняются задним числом.

Текущее бизнес-решение:

- Дефолтный профиль один общий для всех пользователей MVP.
- Изменение дефолта влияет только на новые расчеты, а не на уже сохраненные расчеты.
- Точечные построчные складские наценки и выбранные дополнительные fulfillment-операции пока остаются внутри конкретного расчета; при необходимости их можно добавить в админскую форму следующим этапом.

## Этап 6: полировка и безопасность

Статус: реализована базовая заготовка.

Что добавлено:

- Server action сохранения расчета теперь явно проверяет, что выбранный селлер принадлежит текущему пользователю.
- При обновлении сохраненного расчета action дополнительно проверяет, что расчет принадлежит текущему пользователю и относится к выбранному селлеру.
- Добавлена SQL-миграция `supabase/migrations/202606220004_harden_calculation_seller_ownership.sql`.
- На уровне БД добавлен trigger `calculations_seller_owner_guard`: расчет нельзя привязать к селлеру другого владельца.
- RLS-политики `calculations_insert_own_approved` и `calculations_update_own_or_admin` усилены проверкой связи `seller_id` и `owner_id`.
- Для ошибки доступа при сохранении добавлено понятное сообщение в интерфейсе.
- Добавлен чеклист демо-деплоя `docs/demo-deploy-checklist.md`.
- README теперь ссылается на чеклист и порядок подключения Supabase.
- Добавлен регрессионный тест на наличие ownership guard в Supabase-миграции.

Текущее бизнес-решение:

- Администратор смотрит чужие расчеты через основной экран только в режиме просмотра.
- Пользовательские server actions не позволяют сохранять расчет в чужого селлера даже при ручной подмене hidden-полей формы.
- Финальная живая проверка регистрации, подтверждения пользователя и сохранения расчетов требует подключенного Supabase-проекта.
